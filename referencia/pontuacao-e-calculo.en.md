# Scoring and Calculation of the AI Maturity Assessment

> **Technical reference document**: describes precisely how each response becomes a score, how capabilities/pillars/overall are aggregated, threshold rules, multi-respondent handling, gap analysis and PE score. All formulas match 1:1 the Rust code in `app/backend/src/scoring.rs` (a file in the platform repository, not included in this kit).
>
> **Executable implementations in this kit:** [`scripts/compute_scores.py`](../scripts/compute_scores.py), [`scripts/compute_gaps.py`](../scripts/compute_gaps.py) and [`scripts/recommend_strategies.py`](../scripts/recommend_strategies.py) implement exactly these formulas deterministically. The auditable workbook `pontuacao-e-calculo.xlsx` is generated by [`scripts/generate_scoring_workbook.py`](../scripts/generate_scoring_workbook.py). The real question and capability weights come from [`framework.json`](../framework.json), the single source of truth.

## 1. Conceptual model in 3 layers

```
┌─────────────────────────────────────────────────────────────┐
│                    OVERALL SCORE (0–4)                      │
│   = weighted average of ALL capabilities (not pillars)      │
└─────────────────────────────────────────────────────────────┘
                ▲
                │
┌─────────────────────────────────────────────────────────────┐
│                  PILLAR SCORE (P1, P2, P3)                  │
│        = weighted average of the pillar's capabilities      │
└─────────────────────────────────────────────────────────────┘
                ▲
                │
┌─────────────────────────────────────────────────────────────┐
│                CAPABILITY SCORE (P1-C1 … P3-C9)             │
│          = weighted average of the capability's questions   │
└─────────────────────────────────────────────────────────────┘
                ▲
                │
┌─────────────────────────────────────────────────────────────┐
│              QUESTION RESPONSE (L0=0 … L4=4)                │
│       = level selected by the respondent (multi → average)  │
└─────────────────────────────────────────────────────────────┘
```

**Important characteristic:** the **overall** is computed directly over the capabilities (SUMPRODUCT), it is **not** the average of the 3 pillar scores. This prevents a pillar with few capabilities from weighing as much as one with many.

## 2. How each response becomes a number

| Selected level | Label | Numeric value |
|---|---|---|
| L0 | Initial | **0** |
| L1 | Developing | **1** |
| L2 | Defined | **2** |
| L3 | Managed | **3** |
| L4 | Optimizing | **4** |

The scale is **discrete on input (integer 0–4)** but the aggregations produce **continuous floating-point values (`f64`)**, without rounding. Only the presentation layer (UI/report) decides the visual precision (usually 2 decimal places).

## 3. Official formulas

### 3.1 Capability score
> Reference code: `scoring.rs:205-225` (platform repository)

$$
\text{capability\_score} = \frac{\sum_{q \in \text{answered}} (\text{level}_q \times \text{weight}_q)}{\sum_{q \in \text{answered}} \text{weight}_q}
$$

- If no question of the capability was answered → `capability_score = None` (it does not enter the upper-layer calculations).
- Default weights: **1.0**. Allowed range: **[0.5, 2.0]**.

### 3.2 Pillar score
> Reference code: `scoring.rs:227-247` (platform repository)

$$
\text{pillar\_score} = \frac{\sum_{c \in \text{pillar}} (\text{capability\_score}_c \times \text{weight}_c)}{\sum_{c \in \text{pillar}} \text{weight}_c}
$$

Only capabilities with `score = Some(_)` participate (capabilities without any response are skipped).

### 3.3 Overall score
> Reference code: `scoring.rs:250-263` (platform repository)

$$
\text{overall\_score} = \frac{\sum_{c \in \text{ALL capabilities}} (\text{capability\_score}_c \times \text{weight}_c)}{\sum_{c \in \text{ALL capabilities}} \text{weight}_c}
$$

**Attention:** direct SUMPRODUCT over all capabilities, it is **not** `mean(P1, P2, P3)`.

## 4. Handling missing responses

| Situation | Behavior |
|---|---|
| Question not answered | **Ignored**. Adds neither to `wsum` nor to `wtotal`. No penalty. |
| Capability with no response at all | `score = None`. Does not enter the pillar nor the overall. |
| Pillar with no answered capabilities | `pillar_score = 0.0` (rare edge case). |
| Overall with no answered capabilities | `overall_score = 0.0`. |

> **Golden rule:** "answered questions weigh, missing ones disappear". This encourages the respondent *not to guess* when they do not know: the system only penalizes via `threshold_status`, not via a deflated score.

## 5. Minimum coverage threshold

> Reference code: `scoring.rs:351-359` (platform repository)

| Applicable questions | Status | Behavior |
|---|---|---|
| **≥ 40** | `Ok` | Normal scoring, no warning. |
| **25–39** | `Warning` | Scoring computed, but the report displays the banner "Preliminary result: limited reliability". |
| **< 25** | `Blocked` | On the platform: scoring is **refused**, the API responds 422 `InsufficientData`. In this kit: computed anyway and marked `BLOCKED` (see note below). |

"Applicable" = questions visible to the respondent's configured audience (after the `audience` filter). If a respondent is Backend, Frontend-only questions do not count.

> **Kit vs. platform:** the production platform **refuses** scoring below 25 responses (HTTP 422 `InsufficientData`). This kit, via `scripts/compute_scores.py`, **computes anyway** and marks `threshold_status = BLOCKED` in `scores.json` ("compute-and-mark" behavior). A result marked BLOCKED serves only as a preliminary draft and must not support executive decisions.

## 6. Multi-respondent aggregation

> Reference code: `repos/scoring.rs:354-368` (platform repository)

When more than one person answers the same assessment:

1. For each `question_id`, the system computes **`AVG(selected_level)`** over all respondents who answered that question.
2. That average value (which can be fractional, e.g. 2.67) enters the capability score formula as `level_q`.
3. **There is no per-respondent weight**: every respondent counts equally.
4. **There is no stratification by audience**: if Backend and Frontend answer the same Q, the average mixes both.

**Example:** 3 respondents for Q1 with levels 2, 4, 3 → `Q1 = (2+4+3)/3 = 3.0`.

## 7. Maturity labels (score mapping)

> Reference code: `scoring.rs:361-373` (platform repository)

Applied to any score (capability, pillar or overall):

| Score range | Label | Color (token) |
|---|---|---|
| `score < 0.5` | **L0, Initial** | `--color-l0` (red) |
| `0.5 ≤ score < 1.5` | **L1, Developing** | `--color-l1` (amber) |
| `1.5 ≤ score < 2.5` | **L2, Defined** | `--color-l2` (blue) |
| `2.5 ≤ score < 3.5` | **L3, Managed** | `--color-l3` (green) |
| `score ≥ 3.5` | **L4, Optimizing** | `--color-l4` (purple) |

> **Two label taxonomies, both correct:** the scoring JSONs (`saida/scores.json`, `saida/gaps.json`) use the platform's PT-BR labels in the style `L2 — Definido`, the PT-BR equivalents of the labels in the table above. The PDF report catalogs (`relatorios/i18n/{pt-br,en,es}.json`) use the reports' own display naming instead (e.g. `AI-Enhanced` for L2 in the English catalog). Each surface uses its taxonomy: data JSONs follow the platform, PDFs follow the i18n catalog. The numeric cutoffs (0.5 / 1.5 / 2.5 / 3.5) are identical in both.

## 8. Gap analysis and prioritization

> Reference code: `scoring.rs:307-349` (platform repository)

For each capability:

```
target_level   = target_overrides.get(capability_id) or 3.0 (default L3)
gap_size       = target_level − current_score
priority_score = capability_weight × gap_size

If gap_size ≤ 1e-9 (floating-point epsilon) → discard (target already reached)
```

### Priority classification

| `priority_score` | Label | Meaning |
|---|---|---|
| ≥ 2.4 | **P0** | Critical: address within the next 30 days |
| ≥ 1.6 and < 2.4 | **P1** | High: include in the next quarter |
| ≥ 0.9 and < 1.6 | **P2** | Medium: semester backlog |
| < 0.9 | **P3** | Low: monitor |

**Why `weight × gap`?** Illustrative example: capabilities with weight 2.0 and gap 1.5 (priority_score = 3.0) are more urgent than weight 1.0 and gap 2.0 (priority_score = 2.0): the weight reflects strategic impact on the overall. (The real weights of each capability are in `framework.json`.)

## 9. PE Score (Production Engineering Readiness)

> Reference code: `scoring.rs:266-304` (platform repository)

Sub-score computed **only with questions marked `pe = true`** in the seed.

- Filters → recomputes capability/pillar/overall with the subset.
- Same SUMPRODUCT.
- If no question has `pe = true` → returns `None`.
- It is displayed side by side with the general overall, signaling production readiness (resilience, observability, runbooks, SLOs etc.).

## 10. Persistence (tables and materialization)

> Reference migration: `migrations/20260417000000_initial.sql` (platform repository)

| Table | Key columns | When it is populated |
|---|---|---|
| `assessment_scores` | `overall_score`, `pe_score`, `total_applicable`, `total_answered`, `scored_at` | `POST /api/scoring/trigger` |
| `pillar_scores` | `pillar_id`, `score` | idem |
| `capability_scores` | `capability_id`, `score` (NULL if no response), `weight` | idem |
| `gap_analysis` | `capability_id`, `current_score`, `target_level`, `gap_size`, `priority`, `priority_score` | idem |

`GET /api/scoring/results/{assessment_id}` reads **directly from the materialized tables**, it does not recalculate. This guarantees consistency between generated reports and roadmaps.

## 11. End-to-end example: Pillar P1

### Scenario
Capability **P1-C1: AI Coding Assistants** (5 questions). Assessment answered by **2 developers** (R1 and R2). The weights below are the **real weights from `framework.json`**: `P1-C1-Q3 = 0.8`, `P1-C1-Q5 = 1.1`, all other questions `1.0` (default).

### Real responses

| Question | Question (summarized) | Weight | R1 | R2 | **Avg** |
|---|---|---:|---|---|---|
| `P1-C1-Q1` | Adoption of AI code completion tools | 1.0 | L3 (3) | L4 (4) | **3.5** |
| `P1-C1-Q2` | AI for code review and quality improvement | 1.0 | L2 (2) | L3 (3) | **2.5** |
| `P1-C1-Q3` | AI for test generation and maintenance | **0.8** | L1 (1) | L2 (2) | **1.5** |
| `P1-C1-Q4` | Prompt engineering and template management | 1.0 | L2 (2) | L2 (2) | **2.0** |
| `P1-C1-Q5` | Governance and security of AI tools | **1.1** | L3 (3) | L4 (4) | **3.5** |

### Step 1: Capability score (P1-C1)

```
wsum   = (3.5×1.0) + (2.5×1.0) + (1.5×0.8) + (2.0×1.0) + (3.5×1.1)
       = 3.5 + 2.5 + 1.2 + 2.0 + 3.85
       = 13.05

wtotal = 1.0 + 1.0 + 0.8 + 1.0 + 1.1 = 4.9

P1-C1.score = 13.05 / 4.9 = 2.6633 (2.663265…)   →   Label: L3 — Managed
```

The calculation carries full precision (`f64`, no rounding); the values displayed here are rounded to 4 decimal places.

> **Effect of the real weights:** with uniform 1.0 weights the average would be `13.0 / 5.0 = 2.60`. The lower weight on Q3 (0.8, the weakest response) and the higher weight on Q5 (1.1, a strong response) raise the score to 2.6633.

### Step 2: Pillar score (P1)

Suppose P1-C1 is the only answered capability of pillar P1. In `framework.json`, P1-C1 has `weight_capability = 1.2`:

```
ws = 2.663265… × 1.2 = 3.195918…
wt = 1.2
P1.score = 3.195918… / 1.2 = 2.6633   →   Label: L3 — Managed
```

> With a single answered capability, the capability weight cancels out in the division. In a real assessment, P1 has 9 capabilities and the calculation would be a SUMPRODUCT over all that have at least 1 response.

### Step 3: Gap analysis

Default `target_level = 3.0` and `weight_capability = 1.2` (P1-C1's real value in `framework.json`):

```
gap_size       = 3.0 − 2.663265… = 0.336734… ≈ 0.3367
priority_score = 1.2 × 0.336734… = 0.404081… ≈ 0.4041
classification = P3 (Low)   ← because 0.4041 < 0.9
```

### Step 4: Threshold

5 answered questions << 25 → **`threshold_status = Blocked`** if that were the only assessed capability. In production, ≥ 40 answered questions are expected across the whole assessment.

## 12. End-to-end example: Pillar P2

> **Hypothetical example with custom weights.** The 1.5 weights below are didactic, to show the effect of SUMPRODUCT with non-uniform weights. The real `framework.json` uses `weight = 1.0` on **all** questions of P2-C1. The 1.0 capability weights used in Steps 2 and 3 are also illustrative (in the real `framework.json`, P2-C1 has `weight = 1.2`). The allowed range for custom question weights is [0.5, 2.0] (section 3.1).

### Scenario
Capability **P2-C1: CI/CD Pipeline Intelligence** (6 questions). Answered by **1 SRE** + **1 Platform Engineer**. Mix of **hypothetical** weights: Q1 and Q5 with `weight = 1.5` (questions with direct impact on DORA metrics).

### Responses

| Question | Question (summarized) | Weight | SRE | PltEng | **Avg** |
|---|---|---:|---:|---:|---:|
| `P2-C1-Q1` | CI/CD pipeline optimization with AI | **1.5** | L4 (4) | L3 (3) | **3.5** |
| `P2-C1-Q2` | Self-healing builds and auto-correction | 1.0 | L2 (2) | L2 (2) | **2.0** |
| `P2-C1-Q3` | Predictive failure analysis | 1.0 | L1 (1) | L2 (2) | **1.5** |
| `P2-C1-Q4` | Intelligent cache optimization | 1.0 | L2 (2) | L3 (3) | **2.5** |
| `P2-C1-Q5` | DORA metrics and insights | **1.5** | L3 (3) | L4 (4) | **3.5** |
| `P2-C1-Q6` | Automated flaky test triage | 1.0 | L2 (2) | L1 (1) | **1.5** |

### Step 1: Capability score (P2-C1)

```
wsum   = (3.5×1.5) + (2.0×1.0) + (1.5×1.0) + (2.5×1.0) + (3.5×1.5) + (1.5×1.0)
       = 5.25 + 2.0 + 1.5 + 2.5 + 5.25 + 1.5
       = 18.00

wtotal = 1.5 + 1.0 + 1.0 + 1.0 + 1.5 + 1.0 = 7.0

P2-C1.score = 18.00 / 7.0 = 2.5714…   →   Label: L3 — Managed
```

> **Observation:** without the extra weights on Q1 and Q5, the simple average would be `(3.5+2.0+1.5+2.5+3.5+1.5)/6 = 2.4167` → it would drop to L2. The 1.5 weight reflects that these two dimensions matter more for a mature DevOps outcome.

### Step 2: Pillar score (P2) with 2 capabilities

Add P2-C2 (IaC) with score = 1.80, weight = 1.0:

```
ws = (2.5714 × 1.0) + (1.80 × 1.0) = 4.3714
wt = 1.0 + 1.0 = 2.0
P2.score = 4.3714 / 2.0 = 2.1857   →   Label: L2 — Defined
```

### Step 3: Gap analysis (custom target)

For P2-C1, the SRE team set `target_level = 3.5` (above the default):

```
gap_size       = 3.5 − 2.5714 = 0.9286
priority_score = 1.0 × 0.9286 = 0.9286
classification = P2 (Medium)   ← because 0.9 ≤ 0.9286 < 1.6
```

## 13. End-to-end example: Pillar P3

> **Hypothetical example with custom weights.** The 2.0 weights (questions) and 1.5 (capability, in Steps 2 and 3) are didactic. The real `framework.json` uses `weight = 1.0` on **all** questions of P3-C5 and `weight = 1.0` on the P3-C5 capability itself.

### Scenario
Capability **P3-C5: Agentic Applications** (6 questions). Answered by **1 Architect** + **1 ML Engineer** + **1 Security**. **Hypothetical** weights: Q1, Q3 and Q6 with `weight = 2.0` (maximum weights, the innovation frontier).

### Responses

| Question | Question (summarized) | Weight | Arch | ML | Sec | **Avg** |
|---|---|---:|---:|---:|---:|---:|
| `P3-C5-Q1` | Implementation of autonomous AI agents | **2.0** | L2 (2) | L3 (3) | L1 (1) | **2.0** |
| `P3-C5-Q2` | Multi-agent coordination (orchestration) | 1.0 | L1 (1) | L2 (2) | L1 (1) | **1.33** |
| `P3-C5-Q3` | Tool-use and function calling frameworks | **2.0** | L3 (3) | L4 (4) | L2 (2) | **3.0** |
| `P3-C5-Q4` | Persistent agent memory | 1.0 | L2 (2) | L2 (2) | L1 (1) | **1.67** |
| `P3-C5-Q5` | Continuous evaluation and safety guardrails | 1.0 | L1 (1) | L2 (2) | L3 (3) | **2.0** |
| `P3-C5-Q6` | Governance and auditing of agent actions | **2.0** | L1 (1) | L1 (1) | L3 (3) | **1.67** |

### Step 1: Capability score (P3-C5)

```
wsum   = (2.00×2.0) + (1.33×1.0) + (3.00×2.0) + (1.67×1.0) + (2.00×1.0) + (1.67×2.0)
       = 4.00 + 1.33 + 6.00 + 1.67 + 2.00 + 3.34
       = 18.34

wtotal = 2.0 + 1.0 + 2.0 + 1.0 + 1.0 + 2.0 = 9.0

P3-C5.score = 18.34 / 9.0 = 2.0378…   →   Label: L2 — Defined
```

### Step 2: Gap analysis (strategic capability)

Leadership set `target_level = 4.0` (ambition: lead in the agentic space) and the capability has `weight = 1.5`:

```
gap_size       = 4.0 − 2.0378 = 1.9622
priority_score = 1.5 × 1.9622 = 2.9433
classification = P0 (Critical)   ← because 2.9433 ≥ 2.4
```

→ This capability **enters the roadmap for the next 30 days** with maximum priority.

### Step 3: Contribution to the overall

If the complete assessment has 28 active capabilities, P3-C5 with `score = 2.0378` and `weight = 1.5` contributes:
- Overall numerator: `+ 2.0378 × 1.5 = +3.0567`
- Overall denominator: `+ 1.5`

→ Raising P3-C5 from 2.04 to 3.5 (practical target L3) would add `(3.5 − 2.04) × 1.5 = 2.19` to the numerator and raise the overall by ~`2.19 / wtotal_overall` points.

## 14. Edge cases and guarantees

| Situation | Guarantee |
|---|---|
| `wtotal = 0` (no question answered) | Returns `None` (capability) or `0.0` (pillar/overall). Never divides by zero. |
| Score above 4.0 | Impossible by construction: all levels ∈ [0,4] and weighted averages preserve the range. |
| Score below 0.0 | Impossible: `selected_level ∈ [0,4]`. |
| Negative `gap_size` (already past the target) | Filtered out (does not appear in the roadmap). |
| Multi-respondent with 0 responses | Capability becomes `None`, without error. |
| Respondent outside the audience | Their responses to non-applicable questions are **ignored in scoring** but stored for auditing. |
| Reprocessing (recalculating after a new response) | Idempotent: `POST /api/scoring/trigger` replaces the 4 materialized tables in one transaction. |

## 15. Glossary

| Term | Definition |
|---|---|
| **Question** | Concrete assessment item. Standard ID `P[1-3]-C[1-19]-Q[1-99]`. |
| **Capability** | Functional subdomain. Groups 5–7 questions. |
| **Pillar** | Strategic dimension. Groups 9–10 capabilities. P1, P2 or P3. |
| **Level (L0–L4)** | Maturity of an individual response. Integer 0–4. |
| **Score** | Continuous result `f64 ∈ [0,4]` produced by aggregation. |
| **Weight** | Weight of the question (`[0.5, 2.0]`, default 1.0) or of the capability. |
| **Threshold** | Minimum coverage of answered questions: 25 (warning), 40 (ok). |
| **PE flag** | Marks questions critical for Production Engineering. They generate a parallel sub-score. |
| **Gap** | `target − current` per capability. |
| **Priority score** | `weight × gap` that classifies a capability into P0/P1/P2/P3. |
| **Audience** | Target audiences of the question (developer, sre, security…). Filters visibility in the form. |
| **`threshold_status`** | `Ok` / `Warning` / `Blocked`, returned along with the result. |
